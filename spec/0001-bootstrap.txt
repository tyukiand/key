CLI program - porcelain for ssh-generate and ssh-add

written in rust / cargo
Idea: maintains state of ssh keys on one machine
The persistent state is stored inside of ~/.key directory
The transient state is inferred from the calls to ssh-add (for displaying some of the current status, see below)

Settings are stored in ~/.key/settings.json
Settings currently include only known users:
   {
     "users": string[]
   }

SSH keys are stored in ~/.key/keys , one key per subdirectory:

~/.key/keys/key-id-date/
  key     // private key
  key.pub // public key
  info.json: 
    "creation_date": "yyyy-mm-dd_HH-MM_z"
    "password_storage": string // description of where the password for the key is stored / or maybe a hint, free-form text
    "comment" // free-form comment (not same as `-C` in ssh-add, just a comment stored in this file)

date is yyyy-mm-dd_HH-MM_z, like 2034-01-02_23-55_UTC+3 or something like that (pick the more standard one, if needed)
When creating the key, user specifies only key-id part, date is filled out automatically.
It should be straightforward to generate multiple keys with same key-id for different dates, e.g.  key-id="github-private" that one rotates every couple months.
When activating or deleting or otherwise modifying the key, 

The state of the keys subdirectory and the settings.json can be merkle-hashed into a single sha256 that (almost) uniquely identifies the state of the system.

The hashing must be location-invariant: the "keys_directory" path should be treated as an alpha-invariant name, it should not care whether it's on `/home/bob/.key/keys` or `/home/alice/.key/keys`, only the structure matters.

main commands for maintaining the state:

key user list
key user add name@where
key user delete [name@where] // goes into interactive mode, allows to select by number

Whenever we delete either a key or a user, we must force the user to type out the full name of the deleted entity (non-reversible, no undo)

key list    // lists all existing keys, short
key list -v // list all existing keys, with password_storage notes, comments, creation date
key add [key-id-without-date]  // starts interactively setting up a new key,
  - if not specified as optional part, asks for key-id-wihout-date, checks non-colliding with already existing keys
  - allows to pick a user from list or specify a new user (assume list of users short, show all at once)
  - asks where the password is stored
  - asks for comment (optional, omitted if left empty)
  - automatically fills out date in curnent time zone, with correct time zone offset (pick the simplest thing that works, don't bother with +-15 minutes summer time corrections if that's too hard)
  - the full id becomes key-id-date

key delete [key-id]
  - if optional part not specified, lets pick by number from list, and then forces to re-type in full before deletion of key

Now, the main command is 

key activate [optional key id]

If key id is not given, lets select from list of existing keys.
Then prints the creation date, comment, and the hint about where to get the password.
Then delegates to `ssh-add` to interactively ask for key 

At no point should any passwords actually flow through program's memory, it's always delegated to ssh-gen and ssh-add

The ssh-add works through the $SSH_some_variable that allows to add keys without `eval` ing or `source`-ing anything in the console.

Prerequisites of ssh-add and ssh-keygen must be checked ahead of time, terminate with clear message.

Every command must know whether it's read-only or mutating.
There should be a flag `key --read-only` that guarantees to execute only read-only commands
(every mutating command must receive an explicit MutationToken)

For non-release test-only we need flags:
   --test-only-key-dir <alternative .key dir>  that overrides ~/.key by some temporary directory 
   --test-only-canned-keys <directory with N~=10 examples of canned priv key pub key files previously generated by ssh-keygen and saved for tests, in this mode the key add [blah] just hashes all the inputs, computes mod N, and picks the canned keys and pretends as if that's what has been generated by ssh-keygen)

Finally:

  key status

 should print merkle-hash of current state
 created users
 created keys
 
Also, it should ask ssh-add to get the sha256 hashes to figure out which of the keys are currently activated and which ones are not

General guidelines:
  - operations must be invertible where meaningful (add key -> delete key, add user -> delete user) etc.
  - state must be inspectable (list, status)
  - detailed help
  - extra flags only where really needed
